💡 Simple SELECT Example

Scenario: We have an Employees table with 10,000 rows

Query on HEAP (No Index):

SELECT * FROM Employees WHERE EmployeeID = 5000;

🔴 Table Scan: Database reads ALL pages sequentially until it finds ID 5000
→ Might read 1,250 pages (assuming 8 rows per page)

Query on CLUSTERED INDEX (B-Tree):

SELECT * FROM Employees WHERE EmployeeID = 5000;

🟢 Index Seek: Database traverses B-Tree:
→ Root Node (1 page)
→ Intermediate Node (1 page)
→ Leaf Node with data (1 page)
→ Total: Only 3 pages read!

📖 Key Definitions

🔴 Full Table Scan

Definition: Database reads EVERY row in EVERY page of the entire table sequentially from start to end.

When it happens: No index exists, or query doesn't use indexed columns.
Performance: ❌ SLOW - reads all data even if you need just 1 row
Example: SELECT * FROM Employees WHERE FirstName = 'John' (no index on FirstName)

📦 Heap

Definition: A table with NO clustered index. Data is stored in no particular order.

Characteristics: - Data inserted wherever there's space
- No physical ordering
- Requires full table scan for lookups
Use case: Temporary staging tables or small lookup tables

🟢 Index Seek

Definition: Database uses B-Tree index to jump directly to specific rows without scanning the entire table.

When it happens: Query filters on indexed column with specific values (=, <, >, BETWEEN)
Performance: ✅ FAST - reads only necessary pages
Example: SELECT * FROM Employees WHERE EmployeeID = 100 (EmployeeID is indexed)

🟡 Full Index Scan

Definition: Database reads ALL rows in the index (not the table), but in sorted order.

When it happens: Query needs all rows but can use index to avoid table access
Performance: ⚠️ MODERATE - faster than table scan but slower than index seek
Example: SELECT EmployeeID FROM Employees ORDER BY EmployeeID (index scan, no table access needed)

🗂️ Clustered vs Non-Clustered Index

📚 Clustered Index

Index = Table
Row 1: ID=1, Name=Alice
Row 2: ID=2, Name=Bob
Row 3: ID=3, Name=Charlie
Row 4: ID=4, Name=Diana
✓ Data physically sorted by key
✓ Leaf nodes = actual data rows
✓ Only 1 per table
✓ Usually on Primary Key
✓ Fastest for range queries

🔖 Non-Clustered Index

Separate Index Structure
Alice → 📍Row 1
Bob → 📍Row 2
Charlie → 📍Row 3
Diana → 📍Row 4
↓ Points to actual data ↓
✓ Separate structure from table
✓ Leaf nodes = pointers to data
✓ Can have many per table
✓ Extra lookup step needed
✓ Good for specific columns
📌 Key Difference: Clustered index IS the table (sorted), Non-clustered is a copy pointing to the table.

🎯 How SELECT Works: Two Main Scenarios

Scenario 1: Selecting Clustered Index Column

  • No extra lookup needed!

💡 Pro Tip: For best performance, include commonly queried columns in the SELECT list as "INCLUDED COLUMNS" in non-clustered indexes to avoid key lookups. This is called a "covering index."

📝 Quick Summary

  • Page: 8 KB block - basic storage unit
  • Extent: 8 pages grouped together (64 KB)
  • Heap: Unordered table - data everywhere
  • Nodes: B-Tree structure - organized for speed

💡 Key Takeaway: Indexes (B-Trees with nodes) dramatically improve query performance by organizing data, while heaps store data without order, requiring full scans for lookups.
